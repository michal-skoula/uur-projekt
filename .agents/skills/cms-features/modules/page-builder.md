# Page Builder

PageBuilder is the CMS subsystem. A `Page` model owns a JSON `content` column holding an ordered list of sections. Each section is a `{ type, data }` pair. The registry in `config/page-builder.php` maps each `type` slug to two handler classes: a **Schema** (Filament admin form) and a **Template** (frontend renderer).

This doc covers building and maintaining sections — the only part of PageBuilder an agent should touch.

## What a section is

A section has three parts. That's all you need to know:

1. **Schema** — the Filament form the admin fills out (`getSchema()` returns form fields)
2. **Template** — transforms raw CMS JSON into clean view variables (`prepareData()`)
3. **View** — the HTML the visitor sees (Blade file)

You don't need to understand how sections are rendered. The CMS handles routing, hydration, and rendering internally. Just fill out the schema, the template, and the view.

## Scaffolding a new section

```bash
php artisan make:cms-section SectionName --type=blade|livewire|none --no-interaction
```

Arguments and flags:
- `SectionName` — PascalCase name (e.g. `Hero`, `NewsFeature`)
- `--type` — `blade` (default choice), `livewire`, or `none`
- `--test` — generate a Pest test file (livewire type only)

What gets generated per type:

| Type | Files created |
|------|--------------|
| `blade` | Schema, Template class, Blade view |
| `livewire` | Schema, Livewire component, Livewire Blade view |
| `none` | Schema only (no frontend, API/admin-only) |

The command also registers the section in `config/page-builder.php` automatically.

### When to use blade vs livewire

**Blade is the default.** Use it for everything that doesn't need client-side reactivity.

Livewire is only for sections that need:
- Reactive state (filters, toggles, live search)
- Repeated server hydration (polling, real-time updates)
- Complex client-side interactions that go beyond what Alpine.js in a blade view can handle

If you're unsure, use blade. You can always migrate to livewire later.

## Where section files live

| What | Path |
|------|------|
| Schemas | `app/Filament/PageBuilder/Sections/{Name}SectionSchema.php` |
| Blade templates | `app/View/PageBuilder/Sections/{Name}SectionTemplate.php` |
| Blade views | `resources/views/page-builder/sections/{slug}.blade.php` |
| Livewire components | `app/Livewire/{Name}.php` |
| Livewire views | `resources/views/livewire/{kebab-name}.blade.php` |
| Livewire tests | `tests/Feature/Livewire/{Name}Test.php` |
| Registry | `config/page-builder.php` |

## Schema guidelines

The schema defines what the admin sees in the CMS page editor.

- Always set a meaningful `getIcon()` — pick a Heroicon that represents the section's purpose, not the default smiley
- Always set a descriptive `getLabel()` — this is what the admin sees in the section picker
- Use `search-docs` with `packages: ['filament']` before building schemas to find the right form components
- Use Filament's layout components (`Section`, `Grid`, `Tabs`) to organize complex forms — remember they need explicit `->columnSpanFull()` or `->columns(n)` to span full width

## Template guidelines — `prepareData()`

Both blade and livewire templates implement `prepareData(array $data): static`. This is where raw CMS JSON gets transformed into clean, named view variables.

**Never pass raw JSON straight through.** Always explicitly map fields:

```php
public function prepareData(array $data): static
{
    $this->heading = $data['heading'] ?? null;
    $this->body = $data['body'] ?? null;
    $this->ctaLabel = $data['cta']['label'] ?? null;
    $this->ctaUrl = $data['cta']['url'] ?? null;

    return $this;
}
```

**Always use `?? null` fallbacks.** Stored pages may predate new fields — a section saved before a field was added won't have that key in its JSON.

**Filament nested field naming.** A form field named `cta.label` serializes to `$data['cta']['label']`.

## Frontend guidelines

- Every section's root element must be a `<section id="{slug}">` — this is generated by the stub
- Use semantic HTML (`<article>`, `<nav>`, `<figure>`, `<header>`, etc.)
- Ensure basic accessibility: alt text on images, proper heading hierarchy, sufficient color contrast
- Invoke `/tailwindcss-development` skill for all styling work
- Build mobile-first, then add responsive breakpoints

## Conventions

- **Blade view naming.** The view path mirrors the slug: slug `pull-quote` -> `resources/views/page-builder/sections/pull-quote.blade.php`
- **One section per feature.** Don't cram unrelated content types into a single section with conditionals. Make a new section instead.

## Using the config registry

`config/page-builder.php` is the source of truth for all registered sections. Read it to understand what exists:

```bash
php artisan config:show page-builder
```

Or read the file directly at `config/page-builder.php`.

The config has three keys:

- **`sections`** — map of slug to `{ schema, template }` class pairs. This is the full list of every section the CMS knows about. Check here first when looking for a section, understanding what's registered, or debugging "section not found" issues.
- **`disabled`** — array of slugs. Killswitch. Hidden from the admin picker AND skipped at render time. Use when a section has a breaking bug or security issue and needs to be taken offline immediately without deleting stored content.
- **`deprecated`** — array of slugs. Hidden from the admin picker, but existing pages continue to render. Use when a section has been superseded and shouldn't be added to new pages.

### When to disable vs deprecate

- **Disable** — something is broken or dangerous right now. Effect is immediate: the section vanishes from both admin and frontend. Add the slug to the `disabled` array with a comment explaining why.
- **Deprecate** — the section works fine but has been replaced by a better one, or is no longer being used and is due for disabling or deletion. Existing pages keep rendering it, but admins can't add it to new pages. Add the slug to the `deprecated` array with a comment explaining the replacement.

Both lists take slug strings: `['hero', 'old-quote']`. Always leave a comment on the line explaining the reason.

### Do not manually add to the `sections` array

The `sections` array is managed by the `make:cms-section` scaffolder. Do not add or reorder entries by hand — use the command to register new sections. You may only remove entries from it if deleting the entire section, including all files related to it. see the [deletion policy](#deletion-policy) section for more details.

## Gotchas

- **File uploads need public visibility.** Filament defaults to private. Add `->visibility('public')` on `FileUpload` when the frontend needs to read the file.
- **Layout components don't span full width by default.** `Grid`, `Section`, `Fieldset` need explicit `->columnSpanFull()` or `->columns(n)`.
- **Never rename a registered slug.** Slugs are persisted in every page's `content` JSON. Renaming a slug orphans all existing content using it. If a rename is absolutely unavoidable, you must update every reference in every page's `content` column — but this is a last resort that requires explicit approval. Always prefer deprecating the old section and creating a new one instead.

## Boundaries — what you can and cannot touch

### Allowed

- Create and modify section files: schemas, templates, views, tests (paths listed above)
- Read `config/page-builder.php` to check registered sections
- Edit the `disabled` and `deprecated` arrays in `config/page-builder.php` (with a comment explaining why)
- Read contracts and concerns to understand interfaces (but not modify)

### Off-limits — do not modify

- `app/Contracts/SectionSchema.php`, `app/Contracts/SectionTemplate.php`
- `app/Concerns/BuildsSectionSchema.php`
- `app/Helpers/PageBuilderHelper.php`
- `app/Http/Controllers/PageBuilderController.php`
- `app/Console/Commands/MakePageBuilderSectionCommand.php`
- `config/page-builder.php` `sections` array — the scaffolder writes to this; manual edits are off-limits. (`disabled` and `deprecated` arrays are editable)
- `stubs/page-builder/*`

### Deletion policy

- **Do not delete any files without explicit approval.**
- You may only create and modify section files (schemas, templates, views, tests).

## Workflow

This is a new CMS being built incrementally. The page builder internals (contracts, helpers, rendering pipeline, scaffolding command) are still being refined by a human. Sections are the stable, well-defined surface — they plug into a system that's designed for them.

The PR-only workflow exists because:
- **The internals aren't battle-tested yet.** An agent modifying the rendering pipeline or contracts could introduce subtle bugs that break every section, not just the one being worked on. Section files are isolated — a broken section only breaks that section.
- **We need to validate the agentic workflow itself.** We're learning what agents can reliably do in this codebase. Starting with tight constraints and loosening them as confidence grows is safer than starting loose and discovering problems in production.
- **Sections are self-contained.** A section's schema, template, and view don't affect other sections. This makes them ideal for autonomous work — low blast radius, easy to review, easy to revert.

If a rule feels wrong for your specific situation, that's useful signal — explain why in your PR or blocker. The constraints will evolve as the system matures.

### Branch naming

```
section/{slug}
```

Example: `section/hero`, `section/news-feature`

### Verification checklist

Run all three before pushing. Non-negotiable:

```bash
vendor/bin/pint --dirty --format agent
composer analyze
php artisan test --compact
```

All must pass. Do not push with failures.

### Pull request format

Every PR must follow this format:

```markdown
## What this section does
[One paragraph explaining the section's purpose and where it's used]

## Schema fields
[List of admin-facing fields with their types and purpose]

## Frontend behavior
[What the visitor sees, responsive behavior, any interactivity]

## Screenshots
[Desktop and mobile screenshots if available — see below]
```

If the Puppeteer MCP server is available, use it to capture screenshots of the section at desktop and mobile viewports and attach them to the PR. If Puppeteer is not available, note "Screenshots: not available — no browser access in this session" in the PR and move on.

### When you're blocked

If you hit a wall — fighting the system, needing to touch off-limits files, making ugly workarounds — **stop**. Do not hack around it.

1. Push your feature branch to remote
2. Open a draft PR with a clear description of the blocker
3. Label the PR `blocked`
4. Explain what you tried, what failed, and what you think needs to change in the CMS internals

A human will review and either unblock you or adjust the internals.
