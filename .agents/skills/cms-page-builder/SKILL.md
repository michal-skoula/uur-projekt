# CMS Page Builder

A `Page` model owns a JSON `content` column holding an ordered list of sections. Each section is a `{ type, data }` pair. The registry in `config/page-builder.php` maps each `type` slug to two handler classes: a **Schema** (Filament admin form) and a **Template** (frontend renderer).

This skill covers building and maintaining sections only. The CMS handles routing, hydration, and rendering internally.

## What a section is

1. **Schema** ‚Äî the Filament form the admin fills out (`getSchema()` returns form fields)
2. **Template** ‚Äî transforms raw CMS JSON into clean view variables (`prepareData()`)
3. **View** ‚Äî the HTML the visitor sees (Blade file)

## Scaffolding a new section

```bash
php artisan make:cms-section SectionName --type=blade|livewire|none --no-interaction
```

| Argument / Flag | Description                                    |
|-----------------|------------------------------------------------|
| `SectionName`   | PascalCase name (e.g. `Hero`, `NewsFeature`)   |
| `--type`        | `blade` (default), `livewire`, or `none`       |
| `--test`        | Generate a Pest test file (livewire type only) |

| Type       | Files created                                   |
|------------|-------------------------------------------------|
| `blade`    | Schema, Template class, Blade view              |
| `livewire` | Schema, Livewire component, Livewire Blade view |
| `none`     | Schema only (no frontend ‚Äî API/admin-only)      |

The command also registers the section in `config/page-builder.php` automatically.

**Use blade by default.** Use livewire only for sections that need reactive state, real-time updates, or complex client-side interactions beyond Alpine.js. If unsure, use blade.

## Where section files live

| What                | Path                                                        |
|---------------------|-------------------------------------------------------------|
| Schemas             | `app/Filament/PageBuilder/Sections/{Name}SectionSchema.php` |
| Blade templates     | `app/View/PageBuilder/Sections/{Name}SectionTemplate.php`   |
| Blade views         | `resources/views/page-builder.sections/{slug}.blade.php`    |
| Livewire components | `app/Livewire/{Name}.php`                                   |
| Livewire views      | `resources/views/livewire/{kebab-name}.blade.php`           |
| Livewire tests      | `tests/Feature/Livewire/{Name}Test.php`                     |
| Registry            | `config/page-builder.php`                                   |

## Schema guidelines

- Use `search-docs` with `packages: ['filament']` before building schemas to find the right form components.
- Set a meaningful `getIcon()` ‚Äî pick a Heroicon that represents the section's purpose.
- Set a descriptive `getLabel()` ‚Äî this is what the admin sees in the section picker.
- Use Filament layout components (`Section`, `Grid`, `Tabs`) to organize complex forms. These need explicit `->columnSpanFull()` or `->columns(n)` to span full width.

## Template guidelines ‚Äî `prepareData()`

Both blade and livewire templates implement `prepareData(array $data): static`. Never pass raw JSON straight through ‚Äî always explicitly map fields:

```php
public function prepareData(array $data): static
{
    $this->heading = $data['heading'] ?? null;
    $this->body    = $data['body'] ?? null;
    $this->ctaLabel = $data['cta']['label'] ?? null;
    $this->ctaUrl   = $data['cta']['url'] ?? null;

    return $this;
}
```

Always use `?? null` fallbacks ‚Äî stored pages may predate new fields. A Filament field named `cta.label` serializes to `$data['cta']['label']`.

## Frontend guidelines

- Every section's root element must be `<section id="{slug}">` ‚Äî generated by the stub.
- Use semantic HTML (`<article>`, `<nav>`, `<figure>`, `<header>`, etc.).
- Ensure basic accessibility: alt text on images, proper heading hierarchy.
- Invoke the `tailwindcss-development` skill for all styling work.
- Build mobile-first, then add responsive breakpoints.
- Blade view naming mirrors the slug: slug `pull-quote` ‚Üí `resources/views/page-builder/sections/pull-quote.blade.php`.

## Config registry

`config/page-builder.php` is the source of truth for all registered sections:

```bash
php artisan config:show page-builder
```

The config has three keys:

- **`sections`** ‚Äî map of slug to `{ schema, template }` pairs. Managed by the scaffolder ‚Äî do not edit this array by hand.
- **`disabled`** ‚Äî array of slugs. Hides the section from admin picker AND skips it at render time. Use for broken or dangerous sections needing immediate takedown.
- **`deprecated`** ‚Äî array of slugs. Hides from admin picker but existing pages keep rendering. Use when a section has been superseded.

Always add a comment on the same line when disabling or deprecating, explaining why.

**Never rename a registered slug.** Slugs are persisted in every page's `content` JSON. If a rename is unavoidable, every reference in every page's `content` column must be updated ‚Äî this requires explicit approval.

## Gotchas

- **File uploads need public visibility.** Filament defaults to private. Add `->visibility('public')` on `FileUpload` when the frontend needs to read the file.
- **Layout components don't span full width by default.** `Grid`, `Section`, `Fieldset` need `->columnSpanFull()` or `->columns(n)`.
- **One section per feature.** Don't cram unrelated content types into a single section with conditionals ‚Äî make a new section instead.

## Boundaries ‚Äî what can and cannot be touched

**Allowed:**
- Create and modify section files: schemas, templates, views, tests
- Read `config/page-builder.php` to check registered sections
- Edit the `disabled` and `deprecated` arrays in `config/page-builder.php`

**Off-limits ‚Äî do not modify:**
- `app/Contracts/SectionSchema.php`, `app/Contracts/SectionTemplate.php`
- `app/Concerns/BuildsSectionSchema.php`
- `app/Helpers/PageBuilderHelper.php`
- `app/Http/Controllers/PageBuilderController.php`
- `app/Console/Commands/MakePageBuilderSectionCommand.php`
- `config/page-builder.php` `sections` array (the scaffolder writes this)
- `stubs/page-builder/*`

Do not delete any files without explicit approval.

Read `references/workflow.md` before opening a pull request ‚Äî it contains branch naming, PR format, and the verification checklist.

## Error Handling

- If the section doesn't render, verify the slug in `config/page-builder.php` matches the `type` value in the page's `content` JSON.
- If Larastan reports issues after scaffolding, fix all type errors before finalizing ‚Äî do not ignore them.
- If a task requires touching off-limits files, stop and read `references/workflow.md` for the blocked-task procedure.
